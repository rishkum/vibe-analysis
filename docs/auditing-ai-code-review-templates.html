Vibe Coding for Data Analysis

Chapter 7 Auditing AI: Code Review Templates

Because AI models predict plausible code rather than deeply “understanding” business requirements, any output generated by GitHub Copilot must be treated as untrusted by default.

Reviewing AI-generated code requires a fundamental shift in mindset. You must review the behavior, not just the syntax. AI code is usually well-formatted, which can mask dangerous logic.

7.1 The “Two-Gate” Mental Model

When reviewing a script or a Pull Request heavily assisted by AI, separate your evaluation into two distinct gates:

  1. The Run Gate (Safety): Before you even assess if the code is logically correct, you must evaluate its impact alias “blast radius.” Does this code interact with a database? Does it write files to disk? Could it alter external states? Do not execute the code locally until you have cleared it through the Run Gate.
  2. The Ship Gate (Quality): Once deemed safe to run, review the logic, the edge cases, and whether it solves the actual business requirement.

7.2 Demanding Evidence

A common mistake is reading AI code, seeing it has standard comments, and thinking “It looks like it handles null values.”

Do not accept appearance as proof. Demand evidence. If Copilot writes a complex tidyr::pivot_wider() and aggregation pipeline, you or the reviewer must run unit tests or manual verification scripts showing what happens when the data contains NAs.

7.3 The AI Code Review Template

To ensure these best practices are embedded in your team’s workflow, we recommend using a standardized review template. You can copy the following checklist into your Pull Request templates on GitHub, GitLab, or Bitbucket.

7.3.1 Pull Request Review: AI-Assisted Code

1. Safety & Impact Analysis (The “Run Gate”) - [ ] Blast Radius: Does this code modify persistent data, write to a database, or alter systems (e.g., dbWriteTable in R)? - [ ] Data Access: Does this script process sensitive data or PII? If so, does it log or expose it unnecessarily? - [ ] Dependencies: Have any new CRAN or GitHub packages been introduced by the AI? Are they vetted?

2. Functional Correctness (The “Ship Gate”) - [ ] Context vs Assumption: Does this code solve our specific problem, or did the AI hallucinate a simpler problem? Did it use correct business logic for metrics (e.g., how we calculate Revenue)? - [ ] Edge Cases Handle: How does this code handle missing values (NA), empty data frames, or duplicate records? - [ ] Data Loss: Are there any inner_joins or filter steps that silently drop expected data?

3. Architecture & Traceability - [ ] Reinvention: Did the AI reinvent functionality that already exists in our internal packages or utility scripts? - [ ] Evidence: Has the submitter provided unit test results or output logs showing the execution works for boundary cases?

Reviewer Verdict: - [ ] Approve (Changes are safe and logically sound) - [ ] Request Changes (Logic gaps, edge cases missed, or unnecessary abstraction) - Confidence Score (1-5): ______ (How confident are you in the AI’s logic?)

By enforcing this structure, you close the Velocity Gap. You ensure that the speed of vibe coding does not come at the expense of accuracy and reliability in your analytical output.
